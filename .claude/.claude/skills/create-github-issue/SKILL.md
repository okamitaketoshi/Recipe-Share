# GitHub Issue作成スキル

このスキルは、プロジェクト固有のルール（CLAUDE.md等に記載）に従って、GitHub issueを適切な形式で作成します。

## 使用タイミング

以下のようなユーザーの依頼でこのスキルを使用してください：

- 「issueを作成して」「issueを起票して」
- 「〜の作業をissueにして」
- 「必要な作業をissueとして起票して」
- 「〜についてissueを立てて」

## 作成フロー

### 1. プロジェクトルールの確認

まず、プロジェクトのissue作成ルールを確認します：

```bash
# CLAUDE.mdからissue作成ルールを読み込む
cat CLAUDE.md | grep -A 50 "Issue作成ルール"
```

プロジェクトのissue作成ルールには通常以下が含まれます：

- **使用可能なラベル**
- **必須項目**（概要、詳細、技術スタック等）
- **タイトルの形式**
- **その他の制約**

### 2. ユーザーの依頼内容を分析

ユーザーの依頼から以下を抽出します：

- **タスクの種類**: 新機能、バグ修正、環境構築、ドキュメント作成等
- **対象範囲**: 影響を受けるファイル、コンポーネント、システム
- **目的**: 何を実現したいのか
- **制約**: 技術的制約、既知の問題等

### 3. 適切なラベルを選択

プロジェクトで定義されているラベルから、最も適切なものを選択します。

**一般的なラベルの用途**:

| ラベル | 用途 | 例 |
|--------|------|-----|
| 環境構築 | 開発環境・CI/CD・インフラ | Docker設定、Vercel設定、DB環境構築 |
| enhancement | 新機能追加・既存機能改善 | 画像アップロード、検索機能強化 |
| バグ | バグ修正・不具合対応 | レイアウト崩れ、API通信エラー |
| documentation | ドキュメント作成・更新 | README更新、設計書作成 |

### 4. Issue本文を生成

プロジェクトの定型フォーマットに従って本文を作成します。

**標準的なフォーマット** (CLAUDE.mdに記載されている形式を使用):

```markdown
## 概要
（1-2文で何をするissueなのかを簡潔に説明）

## 詳細
- 具体的な実装内容
- 対象ファイル・コンポーネント
- UI/UX要件（該当する場合）
- データベース変更（該当する場合）

## 要検討事項
- 技術選定で検討が必要な点
- 実装方針で判断が必要な点
- デグレリスク・破壊的変更の可能性
- （検討事項がない場合は「なし」と明記）

## 技術スタック
- 使用するライブラリ・フレームワーク
- 新規導入が必要な依存関係
- 関連する既存コード

## 受け入れ条件
- [ ] 機能が正常に動作すること
- [ ] 既存機能にデグレがないこと
- [ ] コードレビューが完了していること
- [ ] テストが追加されていること（該当する場合）
- [ ] ドキュメントが更新されていること（該当する場合）
```

### 5. Issueタイトルを生成

タイトルは以下の特徴を持つようにします：

- **簡潔**: 70文字以内
- **具体的**: 何をするのか明確に
- **絵文字使用可**: 視認性向上のため（任意）

**絵文字の例**:
- 🚀 デプロイ・環境構築
- ✨ 新機能
- 🐛 バグ修正
- 📝 ドキュメント
- 🔧 設定・ツール
- 🎨 UI/UX改善

### 6. gh issue createコマンドを実行

```bash
gh issue create \
  --title "タイトル" \
  --label "ラベル名" \
  --body "$(cat <<'EOF'
本文内容
EOF
)"
```

**重要な注意事項**:
- `--body`は必ずHEREDOCを使用して渡す（複数行対応のため）
- HEREDOCの開始は`<<'EOF'`（シングルクォートで囲む）
- 本文内にバッククォートやドル記号がある場合でもエスケープ不要

### 7. 作成結果を報告

issueが作成されたら、以下を報告します：

- ✅ 作成されたissueのURL
- 📋 issueタイトル
- 🏷️ 付与したラベル
- 📝 本文の要約（簡潔に）

## エラーハンドリング

### gh CLIが利用できない場合

```bash
# gh CLIのインストール確認
which gh || echo "gh CLI is not installed"
```

gh CLIが利用できない場合は、ユーザーに以下を案内します：

```bash
# Homebrewでインストール（macOS/Linux）
brew install gh

# 認証
gh auth login
```

### ラベルが存在しない場合

指定したラベルがリポジトリに存在しない場合、`gh issue create`は失敗します。エラーメッセージを確認し、利用可能なラベルを提案します：

```bash
# リポジトリのラベル一覧を確認
gh label list
```

## 応用例

### 複数のissueを一度に作成

ユーザーが「必要となる作業をissueとして起票して」のように依頼した場合：

1. タスクを分析して、論理的な単位に分割
2. 各タスクの依存関係を特定
3. 適切な順序でissueを作成
4. 各issueに関連issueへの参照を追加（例: `## 関連Issue\n#123, #456`）

### 既存の会話からissueを作成

会話の中で「これissueにして」と言われた場合：

1. 直近の会話履歴から文脈を抽出
2. 実現したい内容、技術的制約、懸念事項を特定
3. 上記の標準フローでissueを作成

## ベストプラクティス

### タイトルの付け方

**Good**:
- 「🖼️ レシピ画像アップロード機能の実装」
- 「🐛 材料検索で全角・半角が区別される問題を修正」
- 「📝 Vercelデプロイ手順書の作成」

**Bad**:
- 「画像」（曖昧すぎる）
- 「レシピに画像を追加できるようにする機能を実装し、Supabase Storageと連携してファイルアップロードAPIを作成し、フロントエンドのUIコンポーネントを追加する」（長すぎる）

### 要検討事項の書き方

単に「検討が必要」と書くのではなく、**具体的な選択肢**を提示します：

**Good**:
```markdown
## 要検討事項
- 画像ストレージ: Supabase Storage vs AWS S3 vs Cloudinary
- 画像サイズ制限: 5MB or 10MB
- サポートする画像形式: JPEG/PNG only or WebP/AVIF対応
```

**Bad**:
```markdown
## 要検討事項
- 画像アップロード機能について検討が必要
```

### 受け入れ条件の書き方

チェックボックス形式で、**客観的に判断可能**な条件を列挙します：

**Good**:
```markdown
## 受け入れ条件
- [ ] 5MB以下の画像がアップロードできること
- [ ] アップロードした画像がレシピ詳細ページに表示されること
- [ ] 画像削除機能が動作すること
- [ ] 画像アップロードのE2Eテストが追加されていること
```

**Bad**:
```markdown
## 受け入れ条件
- [ ] ちゃんと動くこと
- [ ] いい感じになっていること
```

## まとめ

このスキルを使用することで：

- ✅ プロジェクト固有のルールに準拠したissueを自動生成
- ✅ 定型フォーマットの遵守
- ✅ 適切なラベル選択
- ✅ 一貫性のある品質

ユーザーがissue作成を依頼したら、このスキルに従って実行してください。
