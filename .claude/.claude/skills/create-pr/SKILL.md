# GitHub Pull Request Creation Skill

このスキルは、変更内容を分析して構造化されたPR本文を自動生成し、GitHub Pull Requestを作成します。

## 実行フロー

### Step 1: 現在の状態確認

```bash
# 現在のブランチ確認
git branch --show-current

# mainブランチにいる場合はエラー
# PR作成はfeature/fix/chore等のブランチから行う
```

**mainブランチにいる場合：**
ユーザーに「mainブランチからはPRを作成できません。feature/fix/chore等のブランチに切り替えてください」と伝えて終了。

### Step 2: 変更内容の確認

```bash
# mainとの差分統計を確認
git diff main...HEAD --stat

# コミット履歴を確認
git log main..HEAD --oneline

# 詳細な差分を確認（PR本文生成に使用）
git diff main...HEAD
```

**コミットが存在しない場合：**
ユーザーに「mainブランチとの差分がありません。コミットを作成してから再度実行してください」と伝えて終了。

### Step 3: 変更内容の分析

コミット履歴と差分から以下を分析：

1. **変更の種類**
   - 新機能追加、バグ修正、リファクタリング、ドキュメント更新、設定変更など

2. **影響範囲**
   - 変更されたファイル
   - 主要な変更箇所
   - 追加/削除された行数

3. **技術的な詳細**
   - 使用されているライブラリ・フレームワーク
   - 新規導入された依存関係
   - データベーススキーマ変更の有無

4. **目的と背景**
   - なぜこの変更が必要だったのか
   - どのような問題を解決するのか
   - コミットメッセージから背景を抽出

### Step 4: PRタイトル生成

**タイトル生成ルール：**
- 70文字以内
- 絵文字で変更の種類を表現
- 簡潔で内容が分かる

**絵文字の選択：**

| 絵文字 | 用途 | 例 |
|-------|------|-----|
| ✨ | 新機能追加 | ユーザー認証、画像アップロード |
| 🐛 | バグ修正 | レイアウト崩れ、API通信エラー |
| 🔧 | 設定・環境構築 | mise.toml、Vercel設定 |
| 📝 | ドキュメント更新 | README、ガイド作成 |
| 🧪 | テスト追加 | ユニットテスト、E2Eテスト |
| ⚡ | パフォーマンス改善 | クエリ最適化、キャッシュ |
| 🔒 | セキュリティ | 認証強化、脆弱性修正 |
| ♻️ | リファクタリング | コード整理、重複削除 |
| 🎨 | UI/UXデザイン | スタイル変更、レイアウト |
| 🚀 | デプロイ関連 | CI/CD、ビルド設定 |

**タイトル例：**
- `✨ レシピ画像アップロード機能を実装`
- `🐛 検索フィルターのバグを修正`
- `🔧 Supabase Preview環境トリガー用mise task追加`
- `📝 Vercel環境変数設定ガイドを作成`

#### セクションごとの説明

**## 概要**
- 1-3文で簡潔に
- 何をするPRなのかを明確に
- ユーザーやレビュワーが最初に読む部分

**## 背景（該当する場合）**
- なぜこの変更が必要だったのか
- どのような問題があったのか
- コミットメッセージから抽出
- 背景が明確でない場合は省略可

**## 詳細**
- 対象ファイルとその説明
- 具体的な実装内容
- 主要な変更ポイント
- データベーススキーマ変更、API変更など

**## 技術スタック**
- 使用しているライブラリ・フレームワーク
- 新規導入した依存関係
- 関連する既存コード

**## 受け入れ条件**
- チェックボックス形式
- 機能要件、品質要件、レビュー完了など
- レビュワーが確認すべき項目

**## Test plan**
- 実施済みのテスト（[x]でチェック）
- 今後実施予定のテスト（[ ]で未チェック）
- ローカル確認、Preview環境確認など

**## 関連Issue**
- 関連するissueがある場合のみ記載
- `#123`形式でissue番号を記載

**## 優先度**
- 明確な優先度がある場合のみ記載
- High / Medium / Low

#### PR本文生成の考え方

1. **コミットメッセージを活用する**
   - コミットメッセージから背景や詳細を抽出
   - 複数コミットの場合は全体を俯瞰

2. **変更ファイルを整理する**
   - 関連するファイルをグルーピング
   - 主要な変更ファイルを強調

3. **受け入れ条件は具体的に**
   - 抽象的な条件は避ける
   - レビュワーが確認しやすい項目

4. **テストプランは現実的に**
   - 実施済みと未実施を明確に
   - 実施可能なテスト項目のみ記載

### Step 6: PR作成

生成したタイトルと本文で`gh pr create`を実行：

```bash
gh pr create --title "<タイトル>" --body "$(cat <<'EOF'
<生成したPR本文>
EOF
)"
```

**HEREDOCを使用する理由：**
- 複数行の本文を正しく処理
- 特殊文字のエスケープ問題を回避
- マークダウンの書式を保持

**実行例：**

```bash
gh pr create --title "✨ レシピ画像アップロード機能を実装" --body "$(cat <<'EOF'
## 概要

レシピに画像を追加できるようにする機能を実装。

## 詳細
...
EOF
)"
```

## エラーハンドリング

### mainブランチにいる場合

```
❌ エラー: mainブランチからはPRを作成できません

現在のブランチ: main

対処方法:
1. 新しいブランチを作成: git checkout -b feature/xxx
2. または既存のブランチに切り替え: git checkout <branch-name>
```

### コミットが存在しない場合

```
❌ エラー: mainブランチとの差分がありません

対処方法:
1. 変更をコミット: 「コミットしてください」と指示
2. または他のブランチに切り替えてから再実行
```

### gh CLIが利用できない場合

```
❌ エラー: gh CLIがインストールされていません

対処方法:
1. gh CLIをインストール: brew install gh
2. 認証: gh auth login
```

### 既にPRが存在する場合

```
❌ エラー: このブランチからのPRは既に存在します

PR URL: https://github.com/owner/repo/pull/123

対処方法:
- 既存のPRを更新する場合: git pushで変更をプッシュ
- 新しいPRを作成する場合: 別のブランチで作業
```

## 重要な注意事項

### 絶対に守るべきルール

1. **mainブランチからはPR作成しない**
   - 必ずfeature/fix/chore等のブランチから
   - mainブランチにいる場合はエラーを返す

2. **コミット履歴を確認する**
   - `git log main..HEAD`で全コミットを確認
   - 複数コミットの場合は全体を俯瞰してPR本文を生成

3. **PR本文は構造化する**
   - 概要、背景、詳細、受け入れ条件、テストプランを含める
   - レビュワーが理解しやすい形式

4. **HEREDOCを使用する**
   - 複数行の本文を正しく処理
   - マークダウンの書式を保持

### よくある間違い

❌ **mainブランチからPR作成**
```bash
# mainブランチでPR作成（NG）
gh pr create --title "..."
```

✅ **正しい方法**
```bash
# feature/fixブランチに切り替えてからPR作成
git checkout -b feature/xxx
# 変更をコミット
git commit -m "..."
# PR作成
gh pr create --title "..."
```

❌ **PR本文が簡素すぎる**
```bash
gh pr create --title "fix bug" --body "fixed"
```

✅ **正しい方法**
```bash
gh pr create --title "🐛 検索バグを修正" --body "$(cat <<'EOF'
## 概要
...

## 詳細
...

## 受け入れ条件
- [ ] ...
EOF
)"
```

❌ **コミット履歴を確認しない**
```bash
# 最新コミットだけを見てPR本文を生成（NG）
git log -1
```

✅ **正しい方法**
```bash
# mainブランチからの全コミットを確認
git log main..HEAD --oneline
```

## まとめ

このスキルは、変更内容とコミット履歴を分析して構造化されたPR本文を自動生成し、GitHub Pull Requestを作成します。ユーザーが「PR作成してください」と言うだけで、以下がすべて実行されます：

1. ✅ ブランチチェック（mainでないこと）
2. ✅ 変更内容・コミット履歴の確認
3. ✅ 絵文字付きPRタイトル生成
4. ✅ 構造化されたPR本文生成（概要、背景、詳細、受け入れ条件、テストプラン）
5. ✅ `gh pr create`実行
6. ✅ PR URL報告

このワークフローに従うことで、常に高品質で一貫性のあるPRが作成されます。
